# Adjacency: мульти-таргет и стакающиеся эффекты

**Дата:** 2026-05-31
**Статус:** дизайн утверждён

## Цель

Расширить систему Adjacency двумя возможностями:

1. **Несколько пар «таргет → эффект» на одном предмете.** Сейчас предмет имеет
   один `adjacencyTarget` и пачку эффектов, применяемых к нему. Нужно, чтобы
   разные соседи давали разные эффекты.
2. **Стакающиеся / нестакающиеся эффекты.** Эффект может применяться один раз
   (вне зависимости от числа подходящих соседей) либо по разу за каждый
   прилегающий инстанс.

## Формат конфига (`loot.csv`)

Два существующих столбца, `adjacencyTarget` (8) и `adjacencyEffect` (9),
разбиваются символом `|` на пары по индексу.

```
adjacencyTarget:  hat|sheath|frog
adjacencyEffect:  gold:+50%|gold:+50%|gold:-50%
```

Семантика примера: меч получает +50% к золоту рядом со шляпой, +50% рядом с
ножнами, −50% рядом с жабой.

Правила:

- `|` — разделитель **пар**. `target[i]` сопоставляется с `effect[i]`.
- Внутри одного сегмента эффекта сохраняется старый `;` для нескольких эффектов
  на один таргет: `hat` → `gold:+50%;acid:-20%`. (Сейчас не используется, но
  парсер уже умеет — оставляем для совместимости.)
- `*` в конце строки эффекта → **стакающийся**: `gold:+50%*`.
  Без `*` — нестакающийся.
- Звёздочка ставится на каждый эффект отдельно. В `gold:+50%*;acid:-20%`
  стакается только золото.

### Несовпадение количества

Если число сегментов в `adjacencyTarget` ≠ числу сегментов в `adjacencyEffect`,
парсер кидает `FormatException` при загрузке конфига (ошибка видна сразу).

Пустые оба столбца = нет свойства (как сейчас).

## Модель данных

`AdjacencyEffect` (struct) получает поле `Stackable`:

```csharp
public struct AdjacencyEffect
{
    public EffectType Type;     // Gold | Acid
    public float Multiplier;    // +0.5 = +50%
    public bool Stackable;      // '*' в конфиге
}
```

Плоская пара полей в `LootData` заменяется списком правил:

```csharp
public struct AdjacencyRule
{
    public string Target;            // id соседа-триггера
    public AdjacencyEffect[] Effects; // эффекты этой пары
}
```

`LootData`:

- удаляются `AdjacencyTarget` (string) и `AdjacencyEffects` (AdjacencyEffect[]);
- добавляется `AdjacencyRule[] AdjacencyRules` (never null; пустой = нет свойства).

Парсинг пар выполняется новым методом (напр. `AdjacencyRule.ParseRules(targetRaw, effectRaw)`),
который zip-ует два столбца по `|` и переиспользует существующий
`AdjacencyEffect.ParseList` для каждого сегмента эффектов (с поддержкой `*`).

## Логика resolver

`AdjacencyResolver.Resolve` меняет сигнатуру: вместо
`adjacencyTargetOf` + `adjacencyEffectsOf` принимает один
`Func<T, AdjacencyRule[]> adjacencyRulesOf`.

Для каждого предмета:

1. Завести аккумуляторы суммарного множителя по типам:
   `sumGold = 0`, `sumAcid = 0`.
2. Для каждого правила предмета:
   - посчитать `count` = число **различных** соседних инстансов (4-соседство по
     ребру), у которых `id == rule.Target`. (`GetEdgeNeighbors` уже возвращает
     `HashSet` различных инстансов — большой предмет, касающийся двумя клетками,
     считается один раз.)
   - если `count == 0` — пропустить правило.
   - для каждого эффекта правила:
     - `times = effect.Stackable ? count : 1`
     - прибавить `effect.Multiplier * times` к соответствующему аккумулятору
       (`sumGold` или `sumAcid`).
3. Применить **аддитивно и один раз** к базе:
   - `EffectiveGold = round(baseGold * (1 + sumGold))`, не ниже 0.
   - `EffectiveAcid = round(baseAcid * (1 + sumAcid))`, не ниже 1.

Отрицательные стаки суммируются как есть, без клампа вклада на −100%; итог
ограничивается только полом (gold ≥ 0, acid ≥ 1).

### Что НЕ меняется

- Блок исходящего эффекта `NeighborGoldPct` (какашка) — остаётся как есть,
  применяется после, мультипликативно к итоговому золоту соседей.
- `TotalGold` считается после всех эффектов.
- 4-соседство по ребру, self-сосед игнорируется (нужен другой инстанс).

## Изменение поведения (важно)

Текущий resolver применяет эффекты **компаундом** последовательно
(`value *= (1 + mult)`), поэтому `+50%` дважды давали +125%. Новая модель —
**аддитивная**: множители одного типа суммируются и применяются к базе один раз
(`+50% + +50% = +100%`). Конфиги с одним эффектом на таргет (текущие
`shield`, `gem`, `barracuda`, `poop`) ведут себя идентично — изменение видно
только когда на предмет действует 2+ множителя одного типа.

## Тестирование

Логику парсинга и resolver можно проверить вручную через сцену (game jam, CI
нет). Контрольные кейсы:

- одиночная пара (регресс): `shield` рядом с `sword` → +50% (как раньше).
- две пары, оба соседа: меч рядом со шляпой и ножнами → +100%.
- стак: алмаз с `gold:+50%*`, 2 шляпы рядом → +100%; 3 шляпы → +150%.
- нестак: тот же эффект без `*`, 3 шляпы → +50%.
- отрицательный стак: `gold:-50%*`, 3 жабы → −150% → золото = 0.
- несовпадение длин столбцов → исключение при загрузке.
